# Agent Notes

This repository is scaffolded for a Minecraft town mod and related blueprint conversion tools.

## Working conventions

- Start from the knowledge-base map at `docs/ai-kb/INDEX.md`: it indexes the `docs/ai-kb/` technical notes and the `openspec/specs/` capability index, and same-topic docs/specs cross-link with see-also.
- Keep docs in `docs/ai-kb/` concise and factual. When adding a doc, list it in `docs/ai-kb/INDEX.md` and add a see-also link to any same-topic spec in the same change.
- Keep example files small enough to inspect by hand.
- Prefer deterministic converters and validators in `tools/`.
- The `reports/*_library_report.json` files store the compound/building graph in **summary form** (`*_to_summary_dict()`): per-cell lists (`parcel_nodes[].cells`, `building_slots[].footprint`) are folded into `*_count` + `bbox`, and non-volume massing nodes plus node `meta` are dropped — the only report-graph fields any validator reads are `building_slots[].massing_graph.meta.frontage` (cultivation_town), `meta.terrace_levels` (cultivation_sect), and the volume node's `origin/size`, all of which are preserved. The largest report dropped from 164k to 8k lines with this change (`slim-library-reports`). The in-memory `to_dict()` and generation logic are untouched, so `.nbt` output is byte-identical; if you need the full per-cell graph, call `to_dict()` in code or read the `.nbt`.
- Everything under `reports/` is git-ignored as deterministic generator/validator output (regenerate locally as needed, see `docs/ai-kb/09_validation_checklist.md`), **except two hand-curated files that MUST stay tracked** because they are inputs the build cannot re-derive: `reports/town_distinctness_calibration.json` (tuning floors read by `validate_runtime_town_plan.py`) and `reports/cultivation_style_baseline_hashes.txt` (a pre-migration historical hash snapshot). The `.gitignore` enforces this with `reports/*` plus `!`-exceptions for those two — do not delete or git-add the generated reports; let the ignore rule own them.
- Treat `src/main/resources/data/myvillage/` as the current implemented mod data namespace. `minecraft_town_mod` is only a historical placeholder unless a dedicated rename change updates code, resources, specs, and docs together.
- Treat implementation as incomplete until it is reflected in the actual mod resources and the user-facing usage docs. For generated structures, commands, galleries, styles, or gameplay-facing behavior, this means regenerating or updating `src/main/resources/data/myvillage/`, building the mod jar when practical, and updating `README.md` command/usage instructions in the same change.
- Add new settlement families through `tools/buildgen/groups.py`: each group binds a style profile, archetype roster, layout strategy, and scale parameters. Avoid detecting families by matching `style_id` prefixes.
- Generators and validators take a modset `--profile {vanilla,full}` (default `full`, matching the shipped artifacts). `full` keeps confirmed external-mod ids; `vanilla` drops them to `minecraft:` fallbacks on generation and forbids external mod ids on validation. The shipped `myvillage:` self-namespace is always legal because the mod owns those resources; optional external namespaces are still profile-gated. Both generation and validation resolve legal namespaces and mod block ids from one source, `tools/buildgen/modset.py`, over `exmod/mod_block_catalog.json` — do not hardcode mod ids or namespaces in a validator. The shipped tree is `full`; prove external-mod-free output with `--profile vanilla`.
- Add new roof or motif forms through the registries in `tools/buildgen/ops.py`, then list them in style `allowed_roof_types` or `allowed_motifs`. Do not add new form dispatch by string-matching inside passes.
- Add or recalibrate regions (洲/域) and topology rules through the JSON catalog + ruleset under `src/main/resources/data/myvillage/worldgen/` (`region_profile/*.json` + `region_topology.json`), with the shared `tools/buildgen/region_topology.py` as the single source for generation and validation. Do not branch region behavior in code: a region's tier/qi/danger/role/admitted_subjects and the ruleset's count/tier-step/separator palette/role rules are data. The offline generator's contract (no world writes, no biome takeover, no chunk-gen) is unchanged. As of `add-region-runtime-binding`, a passive **runtime companion** (`com.example.myvillage.region.runtime`) consumes the same graph in-game: 中州 center = world origin; world spawn = lowest-tier eligible region (deterministic sort key `(assigned_tier ASC, distance_from_anchor DESC, qi_midpoint ASC, id ASC)`, no-override for existing custom/admin spawns, one-time per world via `SavedData`); the rung API exposes **sets** at tier ties (`next_rung_regions` — do NOT collapse to singletons in code; the 正道/魔道 alignment system resolves ties later); and the runtime is passive (no terrain/biome/chunk-gen writes; the sole world write is the one-time `setDefaultSpawnPos`). Python↔Java parity is enforced by golden fixtures under `src/test/resources/region_runtime_fixtures/` (regenerate via `tools/buildgen/tests/generate_region_runtime_fixtures.py`); the runtime placement scale is `RADIUS_WORLD(4000) / RADIUS_GRAPH_OUTER(1.45 = 1.4 walled ring + 0.05 max embed jitter)`. See `docs/ai-kb/13_region_topology.md` and the `region-runtime-binding` spec.
- Cultivation form vocabulary currently includes `sweeping_eave_roof`, `hip_roof`, `pyramidal_roof`, and the redefined curved `tiered_eave_roof`, plus slot-resolved `COLUMN`, `PLATFORM_STONE`, `RIDGE_ORNAMENT`, and `BALUSTRADE` details. The flying-eave curve is generated from stair/slab geometry and does not depend on an unstaged Asian-decor curved-roof mod; optional external blocks are skins through slots only.
- The `chinese_courtyard` family reuses the cultivation massing slots `PLATFORM_STONE` and `COLUMN` with a vanilla vernacular palette. Add new vernacular roof forms through `ROOF_REGISTRY` and style `allowed_roof_types`; do not dispatch them by string-prefix matching.
- The courtyard/mansion ground + path layer is a **two-axis surface-zoning model** (shipped by `path-surface-zoning`, see also `courtyard-ground-layer` + `courtyard-path-network`). Material is owned by the **zone** (a space property); shape is owned by the **route** (a geometry property). Six zones map to six style slots:
  | surface | zone | layer | slot |
  |---|---|---|---|
  | 规整青石路 (formal axis) | 中轴通途 | path | `PATH_FORMAL` |
  | 天井灰砖铺地 | 天井/院心 | ground | `GROUND_YARD_HEART` |
  | 廊下木石地面 | 廊道 | ground | `PATH_GALLERY` |
  | 白墙夹道 | 夹道 | ground | `PATH_ALLEY` |
  | 苔石曲径 | 游园 | path | `PATH_TOUR` |
  | 水边石阶与小桥 | 水岸 | path | `PATH_WATERSIDE` |
  `_place_yard_ground` classifies every non-building parcel cell (露天 grass via `GROUND_YARD_OPEN`; eave-drip ring → `GROUND_YARD_HEART`; covered-gallery cells → `PATH_GALLERY`; 倒座 夹道 → `PATH_ALLEY`; residual eave → `GROUND_YARD_UNDER_EAVE`) via `_ground_zone_slot`, which **falls back to `GROUND_YARD_UNDER_EAVE` when a style lacks the finer slot** — so `cultivation_town.json` (small-courtyard, no new slots) regenerates byte-identical and stays out of the surface-zone families. `_route_complete_path` paves the **formal backbone** through `PATH_FORMAL` (single-source BFS `pred` tree from the street-gate entry, each door/water/planting/moon-platform endpoint traced back to the gate — NEVER multi-source, which degenerates into disconnected endpoint clumps that never cross the plinth); `_route_tour_path` lays a **winding tour polyline** (假山南 → nearest pond shore → 亭) through `PATH_TOUR` (`minecraft:mossy_stone_bricks`), each segment a single-source shortest-path tree with an obstacle set forcing any segment that would cut through the rockery/pond to route around it; `PATH_WATERSIDE` writes `stone_brick_stairs` descending to the waterline + a slab bridge spanning the pond's narrowest crossing (the deleted 汀步 `rockery_block` spike-row is NOT restored). A multi-source BFS still runs, but only for the `endpoint_unreachable` reachability check. `_place_band_transition_stairs` writes one `stone_brick_stairs` at each plinth-boundary pair on the backbone (only when `plinth_h >= 2`; Δy=1 is a free Minecraft autostep). The **`moon_gate_passage`** parcel is the material boundary: a voxel-walkable 穿墙通道 through the garden screen wall (月洞门 motif on the surrounding cells), with the tour's first waypoint inside the 花园 so formal/tour overlap is zero cells by construction. Adding a new ground/path surface goes through a slot addition in the style JSON (`chinese_mansion.json` carries all six; `chinese_courtyard.json` carries formal + heart + gallery + alley; `cultivation_town.json`/`cultivation_sect.json` carry none and stay byte-stable) — do not hardcode ground or path block ids in `compound.py`. `validate_mansion` enforces `surface_zone_material:<zone>:<cell>`, `tour_segment_disconnected:<from>-><to>`, and `waterside_bridge_incomplete:<first|last>` on top of the voxel-walkability checks; `validate_compound` / `validate_small_courtyard` enforce `endpoint_unreachable`, `ground_layer_hole`, `ground_kind_mismatch`, `plinth_edge_missing_stair`, `path_overlaps_building_door`. The byte-stability guard in `tools/buildgen/tests/test_chinese_courtyard_regression.py` keeps `cultivation_sect_*` and `medieval_*` byte-identical while the surface-zone families (`chinese_mansion_*`, `chinese_courtyard_*`, `cultivation_town_*`) may regenerate.
- **Voxel-walkability validator** (shipped by `rebuild-jiangnan-mansion`): `_voxel_walk_bfs` performs 3D BFS with autostep-up-1 / free-fall from the gate-entry standable column and checks that all building door positions and path endpoints are reachable. This replaces the old 2D graph check for both `chinese_courtyard` and `chinese_mansion`. Error codes: `voxel_unreachable_door:<archetype>`, `voxel_unreachable_endpoint:<x,z>`, `voxel_step_cliff:<a>-><b>`, `voxel_blocked_by_solid:<x,z>`. Both `validate_compound` and `validate_mansion` run the BFS; `validate_small_courtyard` also runs it but skips plinth-cliff checks (no plinth). `validate_compound`'s `plinth_edge_missing_stair` check fires only when `plinth_h >= 2` (Δy=1 is a free Minecraft autostep — no stair required).
- **`chinese_mansion` compound family** (shipped by `rebuild-jiangnan-mansion`, enclosure skeleton rebuilt by `rebuild-mansion-enclosure-plan`): 3-进 江南大宅 with its own validator (`validate_mansion`) and generation function (`generate_mansion`). It is separate from `validate_compound` (which is designed for 1-进 北京四合院 and would fail mansion checks). Six NBTs: `chinese_mansion_001..006.nbt`, generated by `generate_compound_library.py --group chinese_mansion --count 6 --base-seed 20260618 --profile vanilla` (then `--profile full` for shipped artifacts). The mansion now uses an **enclosure-planning model** rather than fixed z-band placement: a real `gate_house` through-building straddles the south perimeter, buildings are placed as orientation-bearing masses, yards are derived as enclosed negative space, and every door-front is a path endpoint. Facing is implemented by door-wall relocation only (`facing=south/north/east/west` → `front/back/east/west` door wall) without rotating footprints or roof axes; form rules are gate_house/front_row north, open_hall south, west_side_wing east, east_side_wing west, tower_house north. `validate_mansion` enforces gate_house presence, form-rule facing, door-on-path, derived-yard adjacency, silhouette spread, and voxel walkability. `chinese_courtyard` / embedded `small_courtyard` still use the older band-coupled planner pending the deferred propagation change. Sub-building archetypes: `open_hall` (敞厅, `FACADE_OPEN` slot), `front_row` (倒座), `side_wing` (厢房), `gate_house` (门屋), `tower_house` (绣楼/藏书楼, `stories=2`), `garden_pavilion` (亭, `chinese_round_ridge`). Garden elements: `garden_rockery` (假山, `myvillage:rockery_block`), `garden_pond` (水池, freeform noise shoreline), `moon_gate_passage` (月洞门穿墙通道 + 花园照壁, the tour-route material boundary), shoreside `covered_gallery` variant (水边廊 along the pond shore), `service_house` (仆役房 archetype, placed along the 倒座 夹道 when the lot has room), and the `PATH_WATERSIDE` 花园 cross-pond crossing (`stone_brick_stairs` descending to the waterline + slab bridge to the 亭/island; the deleted 汀步 `rockery_block` spike-row is NOT restored). `generate_all_structures.py` generates the mansion group via `--mansion-count / --mansion-base-seed`. Style file: `tools/buildgen/styles/chinese_mansion.json`.
- **Mod-decor-block protocol** (`myvillage:` self-namespace): `myvillage:rockery_block` is registered in Java (`ModBlocks.java` / `RockeryBlock.java`) and shipped with the mod. It carries `variant`, `facing`, and `moss_level` blockstate properties. Unlike external mod decor (fetzisdisplays, supplementaries, etc.), it does NOT need a `mod_block_fallbacks.json` entry and is accepted under both `vanilla` and `full` profiles. When adding new `myvillage:` blocks, register them in `ModBlocks.java`, add a `blockstates/*.json`, add model JSON files, and update `validate_mod_block_fallback.py` to accept the new id under both profiles.
- Cultivation settlement composition currently has three realized subjects (see-also specs: `sect-compound-layout`, `sect-worldgen-structure`, `town-districts`):
  - **Sect compound** — `cultivation_sect` compounds are terraced axial complexes realized by `tools/buildgen/sect.py` + `src/main/java/com/example/myvillage/sect/SectGenerator.java`, built by `/myvillage sect [seed]`.
    - Form: an ordered terrace stack (gate / disciple / assembly / scripture / summit, count parametric 4–6) ascending a single fall-line ritual axis from the mountain gate to the cliff-backed principal hall, importance graded by terrace level, mirrored flanks joined by covered galleries (廊), retaining + on-axis stairs between terraces, and an optional per-seed detached-spire flying-bridge (飞桥) feature shipped as three deterministic variants.
    - Contract: the planner exports a terrace profile (elevations/bounds, rise/depth/taper, axis-stair width, cliff-back height) that the sect worldgen consumes to derive the mountain (反推山形).
  - **Sect worldgen** — now implemented; realizer shared with the command through `SectSink`.
    - Structure: a custom `myvillage:sect` `Structure` (`SectStructure`/`SectStructurePiece`/`SectStructures`, biome-gated by `tags/worldgen/biome/has_sect`, spaced by `worldgen/structure_set/sect`, `/locate`-able) sites sects and bakes a terrace-derived mountain — skeleton + seed noise + outer blend skirt + sheer cliff-back + placed cloud-sea (云海面) + solitary peak (孤峰) under the detached volume — via `SectMountain.java` (mirrored/validated offline by `tools/buildgen/sect_mountain.py`).
    - Command: `/myvillage sect worldgen [seed] [variant]` force-generates a worldgen-style sect with the derived mountain. The solitary peak is raised by the worldgen mountain pass, not by the command compound build.
  - **Town** — the runtime cultivation town (`/myvillage town`) is a districted ~160×160 修仙坊市.
    - Shape: the seed independently selects a base wall family (`square`/`circle`/`oval`/`dshape`/`octagon`/`trapezoid`), an optional `barbican`/`bastion`, and bounded orthogonal grid offsets through the shared Python/Java `town_hash`; outer districts clip to the perimeter while `civic_core` remains rectangular.
    - Districts: gate/market/residential/civic_core/fringe, each binding density, storey band, material register, and archetype roster from the group's `district_brief`. The ritual axis (plaza/paifang/lanterns) sits in the civic core; market/residential parcels carry street frontage (party-wall rows, shared gables, narrow alleys); the civic core carries vertical-landmark archetypes (`pagoda`/`pavilion`/`bell_drum_tower`) satisfying a skyline relief rule.
    - Library vs canonical: the static `cultivation_town_NNN` compound library is district-fill courtyard tissue — `/myvillage place cultivation_town_001` remains a review fragment, but `/myvillage town` is the canonical cultivation town.
    - Street life: cultivation-themed dressing (幌子 banners, 药圃/灵田 beds, 炼丹炉, 法器摊, 阵纹 floors) with profile-gated external-decor fixtures and scalable villager/spirit-fox inhabitants.
- Keep `CHANGELOG.md` current when preparing a release or accepted fix. The version-bump mechanics (large/small/fix increments and the four files that must move together) are the authoritative rule in `openspec/config.yaml` (`rules.tasks`) — follow it there; do not restate it here.
- For staged manual acceptance, prepare both the buildable mod artifact and the command documentation before asking for visual review. The full generate/validate/preview/build command checklist lives in `docs/ai-kb/09_validation_checklist.md` ("Acceptance / preview command checklist") — run it, then confirm an up-to-date README command list for the `/myvillage` commands.
- Chunky acceptance automation is a staged handoff path, not a visual-review replacement. Stage 1 (`python3 tools/run_chunky_acceptance.py --stage 1`) proves the isolated server + Chunky + RCON lifecycle; Stage 2 (`--stage 2`) additionally drives the coordinate `/myvillage ...at` commands from RCON and writes `reports/chunky_acceptance_report.json`. Later full optional-mod and natural sect worldgen stages build on this report.
- Treat preview generation as an acceptance handoff, not just task completion: ensure the aggregate `out/preview/index.html` exists, serve `out/preview/` over HTTP, and report the host's own address as the review URL. Keep that server running until the user says it can be closed, or until the related OpenSpec change is being archived.
- When commands or acceptance prep steps change, update `README.md`, this `AGENTS.md`, and the relevant `openspec/specs/` documents in the same change.
- Git workflow: branches off `main` for each change; **unless the user says otherwise, fast-forward-merge the change branch back into `main` and push after committing** (do not leave finished work stranded on a side branch). Only keep a branch open (or open a PR) when the user explicitly asks for review first.
