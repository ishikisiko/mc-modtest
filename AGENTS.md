# Agent Notes

This repository is scaffolded for a Minecraft town mod and related blueprint conversion tools.

## Working conventions

- Start from the knowledge-base map at `docs/ai-kb/INDEX.md`: it indexes the `docs/ai-kb/` technical notes and the `openspec/specs/` capability index, and same-topic docs/specs cross-link with see-also.
- Keep docs in `docs/ai-kb/` concise and factual. When adding a doc, list it in `docs/ai-kb/INDEX.md` and add a see-also link to any same-topic spec in the same change.
- Keep example files small enough to inspect by hand.
- Prefer deterministic converters and validators in `tools/`.
- The `reports/*_library_report.json` files store the compound/building graph in **summary form** (`*_to_summary_dict()`): per-cell lists (`parcel_nodes[].cells`, `building_slots[].footprint`) are folded into `*_count` + `bbox`, and non-volume massing nodes plus node `meta` are dropped вАФ the only report-graph fields any validator reads are `building_slots[].massing_graph.meta.frontage` (cultivation_town), `meta.terrace_levels` (cultivation_sect), and the volume node's `origin/size`, all of which are preserved. The largest report dropped from 164k to 8k lines with this change (`slim-library-reports`). The in-memory `to_dict()` and generation logic are untouched, so `.nbt` output is byte-identical; if you need the full per-cell graph, call `to_dict()` in code or read the `.nbt`.
- Everything under `reports/` is git-ignored as deterministic generator/validator output (regenerate locally as needed, see `docs/ai-kb/09_validation_checklist.md`), **except two hand-curated files that MUST stay tracked** because they are inputs the build cannot re-derive: `reports/town_distinctness_calibration.json` (tuning floors read by `validate_runtime_town_plan.py`) and `reports/cultivation_style_baseline_hashes.txt` (a pre-migration historical hash snapshot). The `.gitignore` enforces this with `reports/*` plus `!`-exceptions for those two вАФ do not delete or git-add the generated reports; let the ignore rule own them.
- Treat `src/main/resources/data/myvillage/` as the current implemented mod data namespace. `minecraft_town_mod` is only a historical placeholder unless a dedicated rename change updates code, resources, specs, and docs together.
- Treat implementation as incomplete until it is reflected in the actual mod resources and the user-facing usage docs. For generated structures, commands, galleries, styles, or gameplay-facing behavior, this means regenerating or updating `src/main/resources/data/myvillage/`, building the mod jar when practical, and updating `README.md` command/usage instructions in the same change.
- Add new settlement families through `tools/buildgen/groups.py`: each group binds a style profile, archetype roster, layout strategy, and scale parameters. Avoid detecting families by matching `style_id` prefixes.
- Generators and validators take a modset `--profile {vanilla,full}` (default `full`, matching the shipped artifacts). `full` keeps confirmed external-mod ids; `vanilla` drops them to `minecraft:` fallbacks on generation and forbids external mod ids on validation. The shipped `myvillage:` self-namespace is always legal because the mod owns those resources; optional external namespaces are still profile-gated. Both generation and validation resolve legal namespaces and mod block ids from one source, `tools/buildgen/modset.py`, over `exmod/mod_block_catalog.json` вАФ do not hardcode mod ids or namespaces in a validator. The shipped tree is `full`; prove external-mod-free output with `--profile vanilla`.
- Add new roof or motif forms through the registries in `tools/buildgen/ops.py`, then list them in style `allowed_roof_types` or `allowed_motifs`. Do not add new form dispatch by string-matching inside passes.
- Add or recalibrate regions (жі≤/еЯЯ) and topology rules through the JSON catalog + ruleset under `src/main/resources/data/myvillage/worldgen/` (`region_profile/*.json` + `region_topology.json`), with the shared `tools/buildgen/region_topology.py` as the single source for generation and validation. Do not branch region behavior in code: a region's tier/qi/danger/role/admitted_subjects and the ruleset's count/tier-step/separator palette/role rules are data. The offline generator's contract (no world writes, no biome takeover, no chunk-gen) is unchanged. As of `add-region-runtime-binding`, a passive **runtime companion** (`com.example.myvillage.region.runtime`) consumes the same graph in-game: дЄ≠еЈЮ center = world origin; world spawn = lowest-tier eligible region (deterministic sort key `(assigned_tier ASC, distance_from_anchor DESC, qi_midpoint ASC, id ASC)`, no-override for existing custom/admin spawns, one-time per world via `SavedData`); the rung API exposes **sets** at tier ties (`next_rung_regions` вАФ do NOT collapse to singletons in code; the ж≠£йБУ/й≠ФйБУ alignment system resolves ties later); and the runtime is passive (no terrain/biome/chunk-gen writes; the sole world write is the one-time `setDefaultSpawnPos`). PythonвЖФJava parity is enforced by golden fixtures under `src/test/resources/region_runtime_fixtures/` (regenerate via `tools/buildgen/tests/generate_region_runtime_fixtures.py`); the runtime placement scale is `RADIUS_WORLD(4000) / RADIUS_GRAPH_OUTER(1.45 = 1.4 walled ring + 0.05 max embed jitter)`. See `docs/ai-kb/13_region_topology.md` and the `region-runtime-binding` spec.
- Cultivation form vocabulary currently includes `sweeping_eave_roof`, `hip_roof`, `pyramidal_roof`, and the redefined curved `tiered_eave_roof`, plus slot-resolved `COLUMN`, `PLATFORM_STONE`, `RIDGE_ORNAMENT`, and `BALUSTRADE` details. The flying-eave curve is generated from stair/slab geometry and does not depend on an unstaged Asian-decor curved-roof mod; optional external blocks are skins through slots only.
- The `chinese_courtyard` family reuses the cultivation massing slots `PLATFORM_STONE` and `COLUMN` with a vanilla vernacular palette. Add new vernacular roof forms through `ROOF_REGISTRY` and style `allowed_roof_types`; do not dispatch them by string-prefix matching.
- The courtyard ground + path layer is data-driven through the two specs `courtyard-ground-layer` and `courtyard-path-network` (shipped by `fix-courtyard-ground-walkability`). `_place_yard_ground` fills every non-building parcel cell via the style's `GROUND_YARD_OPEN` / `GROUND_YARD_UNDER_EAVE` slots (йЬ≤е§© grass / paver vs е±Лж™РдЄЛ stone_bricks), and `_route_complete_path` paves a **shortest-path backbone** rooted at the street-gate entry (single-source BFS `pred` tree, each door/water/planting/moon-platform endpoint traced back to the gate) rather than flooding the whole reachable set вАФ only backbone cells become `GROUND_PATH`, the rest keep their grass/stone tile. A multi-source BFS is still run, but only for the `endpoint_unreachable` reachability check; the backbone MUST be single-source (a multi-source predecessor tree degenerates into disconnected endpoint clumps that never cross the plinth, leaving main-yard doors `voxel_unreachable_door`). `_place_band_transition_stairs` then writes one `stone_brick_stairs` at each plinth-boundary pair on the backbone. Adding a new ground-layer block goes through those two slot additions in the style JSON вАФ do not hardcode ground or path block ids in `compound.py`. `validate_compound` / `validate_small_courtyard` enforce the five error codes (`endpoint_unreachable`, `ground_layer_hole`, `ground_kind_mismatch`, `plinth_edge_missing_stair`, `path_overlaps_building_door`); the byte-stability guard in `tools/buildgen/tests/test_chinese_courtyard_regression.py` keeps `cultivation_sect_*` and `medieval_*` unchanged while `cultivation_town_*` (embedded small-courtyards) and `chinese_courtyard_*` regenerate.
- **Voxel-walkability validator** (shipped by `rebuild-jiangnan-mansion`): `_voxel_walk_bfs` performs 3D BFS with autostep-up-1 / free-fall from the gate-entry standable column and checks that all building door positions and path endpoints are reachable. This replaces the old 2D graph check for both `chinese_courtyard` and `chinese_mansion`. Error codes: `voxel_unreachable_door:<archetype>`, `voxel_unreachable_endpoint:<x,z>`, `voxel_step_cliff:<a>-><b>`, `voxel_blocked_by_solid:<x,z>`. Both `validate_compound` and `validate_mansion` run the BFS; `validate_small_courtyard` also runs it but skips plinth-cliff checks (no plinth). `validate_compound`'s `plinth_edge_missing_stair` check fires only when `plinth_h >= 2` (ќФy=1 is a free Minecraft autostep вАФ no stair required).

- **Mod-decor-block protocol** (`myvillage:` self-namespace): `myvillage:rockery_block` is registered in Java (`ModBlocks.java` / `RockeryBlock.java`) and shipped with the mod. It carries `variant`, `facing`, and `moss_level` blockstate properties. Unlike external mod decor (fetzisdisplays, supplementaries, etc.), it does NOT need a `mod_block_fallbacks.json` entry and is accepted under both `vanilla` and `full` profiles. When adding new `myvillage:` blocks, register them in `ModBlocks.java`, add a `blockstates/*.json`, add model JSON files, and update `validate_mod_block_fallback.py` to accept the new id under both profiles.
