# Mod Decor Block Family

## Purpose

This spec defines the common contract for `myvillage:` decorative blocks — mod-owned blocks that ship pre-baked geometry (model + VoxelShape) for sub-block-precision decorative content. The first instance is `myvillage:rockery_block` (假山, see `garden-rockery`); future instances include `myvillage:pond_stone` (池岸石), `myvillage:garden_lattice` (花窗), `myvillage:garden_pavement` (园林铺地), and `myvillage:ridge_ornament` (屋脊吻兽). The contract ensures every decor class follows the same registration, blockstate, asset, and VoxelShape patterns so a new class is a checklist add, not a redesign.

## Requirements

### Requirement: Each decor class is a separate `myvillage:` block id

The mod SHALL register each decor class as its own block id under the `myvillage:` namespace (`myvillage:rockery_block`, `myvillage:pond_stone`, etc.). Two decor classes SHALL NOT be folded into one id with a `family` blockstate property — each class has its own variant catalog and VoxelShape table, and folding them would entangle unrelated blockstate spaces.

#### Scenario: A new decor class is added

- **WHEN** a future change adds a `myvillage:garden_lattice` decor class
- **THEN** the change SHALL register exactly one new block id `myvillage:garden_lattice` via `DeferredRegister.Blocks`
- **AND** the change SHALL NOT modify any existing decor class's block id or blockstate.

### Requirement: Every decor blockstate exposes `variant × facing × moss_level`

Every decor class SHALL expose three blockstate properties: `variant` (class-specific catalog, see per-class spec), `facing ∈ {north, south, east, west}` (vanilla horizontal facing), and `moss_level ∈ {none, light, heavy}` (weathering/aging). The `facing` property SHALL rotate the model via vanilla model x/y rotation (the model itself is authored in one canonical orientation). The `moss_level` property SHALL select a texture overlay (moss_none, moss_light, moss_heavy) applied to the variant's base textures.

#### Scenario: A decor block has all three properties

- **WHEN** a `myvillage:rockery_block` is placed with `variant=peak_03, facing=east, moss_level=heavy`
- **THEN** the blockstate SHALL be valid for `myvillage:rockery_block`
- **AND** the rendered model SHALL be the `peak_03` model rotated to face east
- **AND** the heavy-moss texture overlay SHALL be applied to the base stone textures.

#### Scenario: A class that does not use moss still exposes the property

- **WHEN** a future `myvillage:garden_lattice` is placed (lattice does not naturally collect moss)
- **THEN** the blockstate SHALL still expose `moss_level`
- **AND** the `moss_level` values SHALL all render identically (no overlay) for that class
- **AND** generators SHALL place lattice with `moss_level=none` by convention.

### Requirement: Asset layout follows `blockstates / models / textures` per class

Every decor class SHALL ship its assets under three predictable paths:
- `assets/myvillage/blockstates/<id>.json` — the blockstate file mapping every `(variant, facing, moss_level)` tuple to a model.
- `assets/myvillage/models/block/<id>/<variant>.json` — one model JSON per variant (in canonical facing=north orientation; facing rotation handled by the blockstate file's `x`/`y` rotation).
- `assets/myvillage/textures/block/<id>/<variant>_<moss_level>.png` — base + moss-overlay textures per variant, at 16×16 resolution.

#### Scenario: The asset tree is reproducible from a manifest

- **WHEN** a Python build helper processes the class's manifest
- **THEN** every `(variant, facing, moss_level)` combination referenced by the class catalog SHALL produce a blockstate entry, a model JSON reference, and a texture reference
- **AND** no orphan asset SHALL remain in the tree.

### Requirement: Each decor class registers via `DeferredRegister.Blocks` with a `VoxelShape` table

Each decor class SHALL register its block via `DeferredRegister.Blocks` in `ModBlocks.java`, with a `Block` subclass (or `BlockBehaviour` config) that overrides `getShape` / `getCollisionShape` to return a per-variant `VoxelShape` from a class-specific lookup table. The `VoxelShape` for each variant SHALL be generated by the offline model tool (see per-class spec) and merged to keep the AABB count ≤ 32 per variant (vanilla's soft limit for performant collision).

#### Scenario: A decor block returns a class-specific VoxelShape

- **WHEN** a player collides with a `myvillage:rockery_block[variant=peak_03]`
- **THEN** the collision SHALL use the `peak_03` VoxelShape from the rockery class's lookup table
- **AND** the VoxelShape's AABB count SHALL be ≤ 32.

#### Scenario: A decor class's VoxelShape table is regenerated

- **WHEN** the offline model tool regenerates the class's variant catalog
- **THEN** the Java `VoxelShape` table SHALL be regenerated from the same voxel fields that produced the model JSONs
- **AND** the model and the VoxelShape SHALL agree on which sub-cells are solid.

### Requirement: The `myvillage:` self-namespace is profile-exempt

Every decor block under the `myvillage:` namespace SHALL be legal under both `vanilla` and `full` modset profiles. The validator SHALL NOT require a `minecraft:` fallback for `myvillage:` decor ids, because the mod ships the assets and the player who installs the mod has the block registered. The `--profile vanilla` flag SHALL still resolve decor block ids normally (no substitution), unlike external-mod ids which are dropped to `minecraft:` fallbacks under `vanilla`.

#### Scenario: Vanilla-profile validation accepts a decor block

- **WHEN** the structure validator runs under `--profile vanilla` and inspects a `myvillage:rockery_block` blockstate
- **THEN** the blockstate SHALL be accepted as legal
- **AND** no fallback substitution SHALL be applied.

#### Scenario: A structure NBT containing decor blocks is placed in a modded world

- **WHEN** a structure containing `myvillage:rockery_block` is placed in a world where the mod is installed
- **THEN** the block SHALL be placed and rendered correctly
- **AND** no client-side "unknown block" error SHALL occur.

### Requirement (FUTURE EXTENSION): Additional decor classes follow this protocol

The following decor classes are tracked as future extensions of this protocol. Each SHALL be added in its own change, registering one new `myvillage:` id per class and following this spec's contract:
- `myvillage:pond_stone` — 池岸自然石 (freeform pond-edge stones).
- `myvillage:garden_lattice` — 花窗漏窗 (decorative wall lattices).
- `myvillage:garden_pavement` — 园林铺地 (patterned garden paving).
- `myvillage:ridge_ornament` — 屋脊吻兽 (ridge-terminal ornaments).

These are NOT validated by any current validator and SHALL NOT be implemented in the `rebuild-jiangnan-mansion` change.
