## Why

The region (洲/域) layer shipped in `add-region-topology` is **offline-only**: a per-seed graph generator + validator, with no runtime presence. Its spec (`region-topology`, Requirement: "emits the graph and a visualization offline only") explicitly defers "writing world blocks, taking over biome assignment, or running during chunk generation." The result is that, in-game, the region layer does not exist: the world has no spawn bound to the tier gradient, no notion of "which 洲 am I in," and no path for future runtime consumers (compass / map / alignment / mobility gating) to read the graph. The tier gradient — constructed outward from 中州 — is the natural cultivation progression axis, but nothing in-game consumes it yet.

This change lands the **first runtime exit** of the region layer: it places the generated graph into the world (中州 center at the world origin, all 洲 within a ~4000-block radius), binds world spawn deterministically to the lowest-tier non-walled region, and exposes the per-player "current region / current tier rung / next rung" as a query API for downstream systems to consume. It does **not** deliver the downstream consumers themselves (compass item, alignment system, mobility/flight unlocks) — those remain separate future changes that read this API.

## What Changes

- **Port the region-topology RNG and generator to Java.** `tools/buildgen/region_topology.py` (splitmix64 + tagged FNV-1a, parity-ready per its header comment) gets a Java mirror that reproduces a graph **bit-identically** from the same world seed. This is the forcing function the Python module's docstring anticipated ("parity-ready for a future Java runtime placement-director").
- **Place the region graph into the world.** Map region-graph coordinates to world blocks with the anchor (中州) at `(0, 0)` and a scale that fits all 洲 inside a ~4000-block radius (anchor-centered). The placement is pure coordinate transform — it does **not** write terrain, override biomes, or run during chunk generation. The transform is documented and offline-validatable for parity.
- **Bind world spawn to the lowest-tier eligible region.** On first world load (one-time, server-side, deterministic per seed), compute the spawn region as the non-walled region with the lowest assigned tier (deterministic tie-break by distance-from-anchor desc, qi midpoint asc, id asc), translate to a world block, run a safe-surface search (spiral outward, capped radius, find non-liquid standing block), and call `setWorldSpawn`. Persisted as the vanilla world spawn; subsequent loads do not recompute unless reset.
- **Expose a region-runtime query API.** A runtime service (server-side) answers, for any world block coordinate or player: `region_at(x, z)`, `current_region(player)`, `current_rung(player)`, `next_rung_regions(player)`. The tier rung ladder is derived per-seed from the distinct assigned tiers (ascending, anchor = top); `next_rung_regions` returns the **set** of regions at the next-higher distinct tier (a set, not a singleton, so a future alignment/道倾向 system can resolve ties — e.g. 正道 vs 魔道 branches at a shared tier like 灵岳 + 西漠).
- **Walled regions stay unreachable from this API's progression view.** `next_rung_regions` excludes `walled` regions (魔域) — they are not part of the tier rung ladder by construction (they do not join the 连 spanning tree). Whether a 魔道 path can later reach 魔域 is a future alignment-system decision, not this API's.
- **Mobility / fast-travel / flight gating is explicitly deferred.** This API exposes `current_rung`; the mobility system that consumes it (unlocking run / 缩地 / 御剑 / cross-continent teleport by rung) is a separate future change.
- **Compass / map / "next region" indicator form is explicitly deferred.** This API exposes `next_rung_regions`; the UI form (compass item, map overlay, NPC direction, landmark beacon) is a separate future change.
- **No new data files for the region graph at runtime.** The Java generator reads the same `region_topology.json` ruleset and `region_profile/*.json` catalog already shipped. The scale (radius-4000 mapping) is a single constant, not a new JSON body — defer tunable scale to a later change if gameplay demands.

## Capabilities

### New Capabilities
- `region-runtime-binding`: the runtime companion to the offline region-topology generator — places the per-seed region graph into the world with 中州 at the origin, binds world spawn to the lowest-tier eligible region, and exposes the `region_at` / `current_rung` / `next_rung_regions` query API for downstream consumers (compass / map / alignment / mobility). Does not write terrain, override biomes, or run during chunk-gen.

### Modified Capabilities
- _None._ The offline `region-topology` capability keeps its current contract (the offline generator still does not write blocks); the runtime side is a separate capability that consumes the graph. `validation` gains parity checks but its requirements are unchanged in scope (still covers the offline invariants; parity is a new check on the existing generator).

## Impact

- **Code (Java)**: first runtime presence of the region layer. A new package (e.g. `com.example.myvillage.region.runtime`) with: a Java port of `region_topology.py`'s RNG + generator (parity-tested against the Python output for fixed seeds), a coordinate-transform that maps graph → world blocks, a spawn-binding service (one-time `setWorldSpawn` on world load), and a region-runtime query service. Hooked via a server-side world-load event handler.
- **Code (Python)**: a new `tools/parity_region_topology.py` (or extension of the existing validator) that emits canonical graph + expected-spawn fixtures for fixed seeds; the Java port's tests consume these as golden files.
- **Tools**: existing `tools/generate_region_topology.py`, `tools/validate_region_topology.py`, and `tools/buildgen/region_topology.py` are unchanged in behavior (they remain the offline source of truth); a small export step may be added to dump golden fixtures for Java parity.
- **Data**: no new shipped JSON. The radius-4000 mapping constant lives in Java (single source; documented in `docs/ai-kb/13_region_topology.md` as a see-also).
- **Specs**: new `region-runtime-binding`; `region-topology` unchanged.
- **Docs**: `docs/ai-kb/13_region_topology.md` extended with a "Runtime binding" section (coord-transform, spawn rule, rung ladder, next-rung-as-set, deferred consumers); `docs/ai-kb/INDEX.md` see-also updated; `README.md` gains a `/myvillage spawn` (or similar) query command if a command hook is added; `AGENTS.md` notes the runtime-binding contract (中州 = origin, spawn = lowest-tier, rung API exposes sets not singletons).
- **Version**: feature bump per the `openspec/config.yaml` version rule (`gradle.properties`, `neoforge.mods.toml`, README jar-name examples, `CHANGELOG.md` together).
- **Depends on**: `add-region-topology` (archived; provides the offline generator, ruleset, catalog). **Enables (future, deferred):** compass/map "next region" indicator; 正道/魔道 alignment-based tie resolution at shared tiers; mobility/flight/teleport gating by rung; runtime placement of subjects (sects/towns) into regions; terrain realization of 隔 edges.
