## ADDED Requirements

### Requirement: Qingfeng Sword resources are packaged without changing structure generation
The practical mod jar SHALL contain the Qingfeng Sword item model, original item texture, `en_us` and `zh_cn` display entries, shaped recipe, and vanilla sword-tag membership together with its registered Java item class. These hand-authored resources SHALL not be generated by or change structure NBT generation, building/compound libraries, placement functions, or worldgen resources.

#### Scenario: The practical jar is inspected
- **WHEN** a current build completes
- **THEN** every Qingfeng model, texture, language, recipe, tag, and class entry SHALL be present and internally referenced

#### Scenario: Structure generation runs
- **WHEN** existing structure resources are regenerated before or after the sword addition
- **THEN** the Qingfeng item SHALL not add or rewrite a structure NBT, place/gallery function, structure set, biome modifier, or generated settlement resource

### Requirement: PAL animation JSON ships at the verified client resource path
The practical mod jar SHALL contain the original seven-animation JSON set below `assets/myvillage/player_animations/`. Animation keys, not the source filename, SHALL be the runtime resource ids. MyVillage SHALL not package the PAL dependency classes or jar inside its own output.

#### Scenario: PAL animation resources are listed
- **WHEN** the jar is inspected after `processResources`
- **THEN** the animation JSON SHALL be below `assets/myvillage/player_animations/`
- **AND** its `animations` object SHALL expose every declared MyVillage id

#### Scenario: Dependency contents are searched
- **WHEN** the MyVillage jar is searched for PAL packages or nested third-party jars
- **THEN** no shaded or copied PAL binary content SHALL be present

### Requirement: Acquisition and combat acceptance commands accompany the artifact
README usage and acceptance documentation SHALL include `/give @s myvillage:qingfeng_sword`, configurable mode-key guidance, vanilla/cultivation behavior checks, five-move checks, hitbox/wall/step checks, multiplayer animation/damage checks, lifecycle/persistence checks, and the debug command. Documentation SHALL keep the rideable sword's independent acquisition and controls intact.

#### Scenario: The jar is handed to a reviewer
- **WHEN** current command documentation is read
- **THEN** the reviewer SHALL have one complete ordered Qingfeng/PAL/combat checklist without needing backend CRAFT or OpenSpec commands
- **AND** existing flying-sword and cultivation user commands SHALL remain discoverable
